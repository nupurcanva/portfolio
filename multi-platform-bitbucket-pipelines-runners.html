Multi-platform design for Bitbucket Pipelines runners

Case study: Crawl, walk and then run

üèÉüèª The Problem

Atlassian's Bitbucket Pipelines offers an easy-to-get-started cloud-based CI/CD solution, but over the years many teams have run into issues, like users cannot access resources in their private network that are protected by firewalls, using other platforms like macOS, scalability issues, rate-limiting, compliance and technical limitations of a cloud-based solution.

This hinders them from using Bitbucket Pipelines as their CI/CD solution. Users want access to internal resources to run integration tests against private resources(eg: database, application), deploy to internal infrastructure, use MacOS or windows platform and access hosted secrets manager.

To do this, users have to open a port in their firewall to do a deployment from Bitbucket Pipeline into their private network, which has security implications, and it is a costly affair to move all of their code and CI/CD to a competing tool. It would be solved if Bitbucket allowed them to run pipeline steps in their own infrastructure.

üí°
Research finding: Users cannot use VPN because of security implications, but most users were open to the idea of "Runners" - like tiny software installed on your hardware, which they can self-host.

üí∞ Impact of this problem

Users are using another CI/CD tool. This causes more friction because they have to set up and pay for another tool, and they miss out on the integrated code and CI/CD experience. Some are moving to other platforms (GitHub/GitLab) since both of them provide a solution to the above problem.

‚ö°
The opportunity: Users looked for a predictable bill - using the on-premises would cut their costs

üí• The challenge

Create an easy-to-use feature that supports all platforms (Windows, macOS etc) so users are able to access their protected resources and Bitbucket Pipelines is the tool of choice for CI/CD.

üéΩ What are Bitbucket Pipelines (CI/CD) and Runners?

Imagine a developer writing some code. They push it to the cloud to integrate it with the rest of the code. Every time they push it, it is being tested, verified & integrated into the codebase and delivered to their customers. This is known as CI/CD (Continuous integration and continuous deployment)

Runners are the way we solved the above problem, so imagine providing a way to install a small software on your computer to run all the CI/CD without sharing it with the cloud. Self-hosted runners offer more control of hardware, control of the Operating system and more security and network options than Bitbucket-hosted runners provide for running the CI/CD.

üîç Research hypothesis and method

Our hypothesis is that users want to run pipelines in their private networks and the way they can do that is by self-hosted runners. We need to begin to validate this belief, and this research represents a step toward understanding our customer's requirements and evaluating the solution. We, therefore, seek to:

  • Understand user's requirements, scale and challenges with self-hosted runners
  • Identify the best solution offering by evaluating the design thinking
  • Learn how we can prioritize future iterations for the project

I did 6 1-hour interviews with large and medium-sized customers. The roles of the interviewees were - Ops engineers, embedded systems developers and lead software developers. I compiled the findings and shared them as a Confluence page with the team before we kicked off the design process.

‚ùì Who is using Bitbucket Runners?

The Repository Manager

They could be the Team lead, a System administrator, a senior developer or a developer. This user must have admin access to the repository, as they set up the repository, variables, runners and the architecture for their team.

The Developer

This user utilizes the build setup to run their builds, tests and deployments. They may use the runners to configure new builds, run tests, and are interested in seeing the outcome.

The Workspace Manager

They could be the Team lead, a System administrator, a senior developer or a developer. This user must have admin access to the account/workspace/project. Manages CI/CD for a bigger team or multiple projects.

ü§î Design considerations

Provide updates while maintaining context

Provide developers real-time updates on runners they've subscribed to. Thread subsequent activity instead of starting a new thread each time.

Talk turkey

To begin with, we've chosen a Command-line based configuration model instead of GUI based interface for our developer persona. However, have experiences to make it easy to access the list of available commands and what they do.

Give extra controls in the place where they like to work

Manage or update the runner directly from the terminal

Provide length and breadth

Users should be able to configure macOS, windows or Linux runners, and the design would be able to support all use-cases

Align with Bitbucket and Atlassian Design Guidelines

Align with the management screens and the installation flow

üåÇ Closing notes

The design considerations allowed me to think broadly and apply the design model to future iterations of the runners.

Outside inside

Obtain new viewpoints. Speak to outside users. To get ideas and set your product against others, look at products that are both closely and distantly competitive to yours.

Think long-term scalability

A prescription for short-term success but failure in the long run is to not have a vision for what the product will look like in a few years. It may cause the product to be directed in the wrong direction, and a U-turn may be very costly.

Inclusive design

When creating products, accessibility is a crucial factor. It goes without saying that we should be inclusive of users with disabilities. As a designer, it's important to ensure the basics like maintaining the minimum colour contrasts and focus states while creating mockups.
